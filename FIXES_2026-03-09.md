# 评估系统修复记录 (2026-03-09)

## 修复概览

本次修复解决了评估系统中的5个关键问题，确保评估结果的可信性和准确性。

## 问题1: 评测失败静默降级为高分mock结果 [High]

### 问题描述
- 当评测异常时，系统会调用 `_run_mock_evaluation()` 返回随机0.75-0.95的成功率
- 这会把"系统挂了"伪装成"效果不错"，严重污染评估结论
- 双系统和System1基线都存在此问题

### 修复内容
**文件：`scripts/eval/run_dualsystem_evaluation.py`**
- 删除 `_run_mock_evaluation()` 方法
- 评测失败时明确标记 `evaluation_status: "FAILED"`，成功率记为0.0
- 策略客户端不可用时标记为 `"SKIPPED"`
- 没有有效预测时标记为 `"NO_VALID_PREDICTIONS"`

**文件：`scripts/eval/run_dreamzero_evaluation.py`**
- 同样删除 `_run_mock_evaluation()` 方法
- 添加相同的失败标记机制

**聚合逻辑改进：**
- 过滤失败/跳过的episode：`valid_results = [r for r in results if r.get('evaluation_status') in ['SUCCESS', None]]`
- 统计失败数量：`num_failed`字段
- 如果全部失败，返回None而不是无效统计

## 问题2: 默认参数不是严格口径 [Medium]

### 问题描述
- 默认 `error_threshold=0.5, eval_steps=5` (宽松口径)
- 文档声称的严格协议是 `error_threshold=0.1, eval_steps=10`
- 不传参数时会得到"乐观结果"，不可用于最终对比

### 修复内容
**双方评估器的默认参数改为：**
```python
parser.add_argument("--error-threshold", type=float, default=0.1,
                   help="L2 threshold for success metric (strict protocol)")
parser.add_argument("--eval-steps", type=int, default=10,
                   help="Number of evenly sampled timesteps per episode (strict protocol)")
```

## 问题3: LLM规划时间未被测量 [Medium]

### 问题描述
- DS比S1多了LLM规划步骤，但时间开销未被记录
- `completion_time` 只测量策略推理，不包括LLM调用
- 无法准确评估DS的真实时间代价

### 修复内容
**文件：`scripts/eval/run_dualsystem_evaluation.py`**

添加LLM时间测量：
```python
llm_plan_start = time.perf_counter()
sub_instructions, planner_meta = self.planner.plan(task_description, ...)
llm_plan_time = time.perf_counter() - llm_plan_start
```

返回结果新增字段：
- `llm_planning_time`: LLM规划时间（秒）
- `total_episode_time`: 策略推理总时间 + LLM规划时间
- `policy_inference_time_per_step`: 每步策略推理时间
- 保留 `completion_time`: 向后兼容，等于 `policy_inference_time_per_step`

摘要新增统计：
- `llm_planning_time_per_episode`: {mean, std}
- `total_episode_time`: {mean, std}

## 问题5: 对外报告存在明显冲突 [Medium]
